# 픽셀로지 오너스 마켓 — AI 운영 엔진 전산 설계서

> 이 문서는 심층 탐구(팟캐스트)와 사업 자료에서 정의된 **"AI의 역할"** 을 실제 돌아가는
> 전산 시스템으로 옮기기 위한 **설계(스펙)** 입니다. 코드 구현 전에 시스템 계층 구조 ·
> AI 역할별 계산식 · DB 스키마 · 정산 룰 · 컴플라이언스를 확정하는 것을 목표로 하며,
> `price_trend_formula.md`(가격 추이 계산식) · `#master_plan.md`(트리거 엔진) ·
> `MOA.md`(위·수탁/무보장 원칙)의 규약과 정합하도록 작성했습니다.

---

## What — AI는 "수익을 만들어주는 마법"이 아니라 "실물 유통 운영 엔진"이다

픽셀로지에서 AI는 다음과 같이 정의합니다.

> **픽셀로지 AI는 상품 소싱, 적정 매입, 채널 배분, 가격 조정, 재고 회전, 정산 검증을
> 전산화하여 실물상품 유통 회전율을 높이고 손실 가능성을 줄이는 운영 엔진이다.**

- ❌ **아님**: "AI 투자상품", "가만히 있어도 매달 O% 보장" (→ 유사수신 리스크)
- ✅ **맞음**: 매입 → 입고 → 판매 → 정산 → 재매입이 반복되는 **실물상품 회전 구조** 안에서
  상품별 성과·채널별 효율을 계산해 **운영 효율을 높이고 손실을 줄이는** 의사결정 운영체제

AI가 실제로 하는 일은 크게 세 가지입니다.
1. **좋은 물건을 고른다** (소싱 평가)
2. **얼마나 사야 하는지 계산한다** (적정 매입량)
3. **어디서 · 얼마에 · 언제 팔지 결정한다** (채널 · 가격 · 시점)

---

## Why — 핵심은 "AI 투자상품"이 아니라 "실물 유통 운영 시스템"

이 사업의 본질은 실물상품 회전 구조입니다. AI 단독으로는 아무것도 완성되지 않으며,
반드시 **AI + WMS(창고) + OMS(주문) + ERP + 정산원장 + 앱** 이 하나로 연결되어야 합니다.

```
[외부 데이터]  쿠팡·네이버·11번가·자체몰·검색트렌드·가격·리뷰
      ↓
[수집/정제]    상품명 표준화 · SKU 매칭 · 가격/판매속도/재고/리뷰 정리
      ↓
[AI 분석 엔진] 수요예측 · 마진예측 · 위험도평가 · 적정매입량 · 채널추천 · 가격추천
      ↓
[운영 시스템]  소싱 승인 · 입고 지시 · 창고 배정 · 채널 배분 · 정산 계산
      ↓
[앱/대시보드]  조합원·운영자·MD·창고·정산 담당자가 같은 데이터 확인
      ↓
[단일 원장]    계약·상품·입고·판매·출고·정산·재매입 통합 저장
```

따라서 AI는 "4번 계층(판단)"에만 있는 게 아니라 **데이터 수집·표준화·입고 검수·채널 배정·정산
이상탐지까지 전 구간**에 들어갑니다.

├─────────────────────────────────────┤
│  6. 정산·리포트 계층                  │

├─────────────────────────────────────┤
│  5. 운영 실행 계층                    │

├─────────────────────────────────────┤
│  4. AI 판단 계층                      │
│     수요예측, 가격추천, 위험필터링      │
├─────────────────────────────────────┤
│  3. 데이터 표준화 계층                 │

**AI의 6대 역할**(3절~8절)은 이 계층들을 관통합니다.

| # | AI 역할 | 시점 | 산출물 |
|---|---|---|---|
| 1 | 상품 소싱 평가 | 매입 전 | 상품성 점수 · A/B/C/D 등급 |
| 2 | 적정 매입 수량 산출 | 매입 전 | 안전 매입량 |
| 3 | 채널 배분 | 입고 시 | 채널별 배분율 |
| 4 | 가격 최적화 | 판매 중(매일) | 추천 판매가 |
| 5 | 재고 회전 위험 감지 | 판매 중(상시) | Green/Yellow/Orange/Red |
| 6 | 정산 검증(이상탐지) | 정산 시 | 이상거래 알림 |

## 2) 데이터 계층 — 수집(멀티채널) + 표준화

### 2-1. 수집 대상
- **외부 오픈마켓/포털**: 쿠팡 · 네이버 · 11번가 (가격·판매순위·리뷰·검색트렌드)
- **자체 채널**: 자체몰 · 폐쇄몰 판매/재고/트래픽
- **내부**: 창고(WMS) 입출고, 정산(ERP)

### 2-2. 실연동 (지금 붙일 수 있는 것) — 네이버 데이터랩/쇼핑 (MCP)
검색 트렌드·경쟁가·시즌성은 아래 네이버 MCP 4종으로 **즉시 실연동** 가능합니다.
(쿠팡/11번가는 공식 API/파트너 커넥터 확보 전까지 크롤러 또는 수동 CSV로 대체)

| 단계 | MCP 툴 | 용도 → 매핑 |
|---|---|---|
| ① 카테고리 확정 | `find_category` | `sku_master.category`(한글 키워드) → category code |
| ② 수요 트렌드 | `datalab_shopping_keywords` (+ `datalab_search`) | 최근 12주 클릭/검색 시계열 → 수요점수 |
| ③ 경쟁가 | `search_shop` | `lprice`(최저가)·판매자수 분포 → 마진/경쟁 압력 |
| ④ 시즌성 | `datalab_shopping_category` | 최근 24개월 월별 지수 → 시즌 정점월 |

> **과거 추이 축적**: `search_shop`은 스냅샷만 주므로, 매일 호출 결과를 `market_price` 테이블에
> 적재해 시계열(`market_price[sku_id][channel][date]`)을 자체 축적합니다. 첫 실행일부터 추세가 쌓입니다.
> 데이터랩은 상대지수 → 판단은 항상 "절대량"이 아니라 "기울기/추세"로 합니다.

### 2-3. 표준화 (3계층)
- **상품명 표준화**: 채널별 상이한 상품명 → 정규 상품명 매핑
- **SKU 매칭**: (브랜드+모델명+옵션) → 단일 `sku_id`로 수렴 (동일 실물 = 동일 SKU)
- **가격 정규화**: 채널별 가격을 (수수료·배송비 제외) 순가 기준으로 통일

### 2-4. CSV 업로드 경로
자사 재고/가격 데이터는 CSV 업로드로도 주입 가능해야 합니다. 업로드 시 컬럼 매핑 UI로
헤더가 달라도 `sku_master`/`inventory_lot`/`market_price` 정규 스키마(11절)로 흡수하고,
검증 게이트(`unit_cost>0`, `qty>0`, `condition/risk_type` 허용집합)를 통과 못한 행은 반려합니다.
→ 팟캐스트의 "인코어 24년 소싱망 = 엑싯 가능한 정품만 편입"을 **입력 게이트**로 강제.

## 3) AI 역할 ① — 상품 소싱 평가 ("이 물건, 가져와도 되는가?")

### 3-1. 후보 상품 입력
브랜드 · 모델명 · 수량 · 매입가 · 유통기한/시즌성 · 공급처 신뢰도 · 현재 온라인 최저가 ·
최근 30일 가격 추이 · 리뷰 수 · 평점 · 검색량 · 경쟁 판매자 수 · 반품 위험 · 보관 난이도

### 3-2. 상품성 점수 (점수화 — "좋음/나쁨"이 아니라 숫자)

```
상품성 점수 = 수요점수 + 마진점수 + 회전점수 + 안정성점수 − 위험점수
```

각 하위 점수는 0~100으로 정규화(가중치·상수는 `sourcing_config.json`으로 분리):

```
수요점수  = f(검색량 추세, 리뷰 수, 평점, 카테고리 시즌 적합도)
마진점수  = clip( (시장판매가능가 − 매입가 − 예상비용) / 매입가 / MARGIN_TARGET , 0, 1)*100
회전점수  = f(예상 일판매량, 경쟁 판매자 수)          # 빠를수록 ↑
안정성점수 = f(공급처 신뢰도, 반품 위험 낮음, 보관 용이)
위험점수  = f(가격 하락 속도, 단종/유통기한 임박, 경쟁 과열)   # 감점
```

### 3-3. 등급 판정 & 예시

```
상품명: LG그램 16인치
매입가: 850,000원   시장 판매가능가: 1,400,000원   예상 마진: 550,000원
판매 회전: 빠름     경쟁 강도: 중간     위험도: 낮음
→ AI 판정: A등급 / 우선 매입 가능
```

| 등급 | 의미 |
|---|---|
| A | 우선 매입 |
| B | 조건부/테스트 매입 |
| C | 회전 위험 — 보류 |
| D | 유통기한 임박/고위험 — 매입 금지 |

> 카테고리별(가전·명품·생활잡화·식품·뷰티)로 재판매가·순수익 구조가 다르므로,
> `sourcing_config.json`에 **카테고리별 가중치 프로파일**을 둡니다.

## 4) AI 역할 ② — 적정 매입 수량 산출 ("얼마나 사야 안전한가?")

좋은 상품이라도 많이 사면 위험합니다.

```
적정 매입량 = 예상 일판매량 × 목표 판매일수 × 안전계수
```

**감산 규칙 (하드 룰)**
- 경쟁 판매자 증가 → 감산
- 가격 하락 속도 빠름 → 감산
- 시즌성 상품 → 기간 제한(시즌 윈도우 내 소진 가능량으로 캡)
- 유통기한 짧음 → 강제 감산
- 창고 보관비 높음 → 감산

**판단 예시**
```
A상품: 5,000개까지 매입 가능
B상품: 500개만 테스트 매입
C상품: 가격 좋지만 회전 위험 높음 → 보류
D상품: 유통기한 임박 → 매입 금지
```

> 전자제품(LG그램)은 **가격 하락 속도·모델 단종**을 더 세게, 마스크팩 등 뷰티는
> **유통기한·리뷰·계절성·대량 재고 리스크**를 더 세게 보도록 카테고리 가중치를 반영.

## 5) AI 역할 ③ — 채널 배분 ("어느 채널에 얼마씩?")

```
┌─────────────┐
│ 상품 입고    │
└─────┬───────┘
      ↓
┌─────────────┐
│ AI 채널 분석 │
└─────┬───────┘
      ↓
┌──────────┬──────────┬──────────┬──────────┐
│ 자체몰    │ 오픈마켓  │ 도매특판  │ 오프라인  │
│ 고마진    │ 빠른회전  │ 대량처분  │ 체험판매  │
└──────────┴──────────┴──────────┴──────────┘
```

**채널 마스터(`channel_config.json`)** — 각 채널을 `{현금화속도, 수수료, 마진잠재력, 물량수용, 허용등급}` 벡터로 정의.

| 채널 | 특성 |
|---|---|
| 자체몰(오너스마켓) | 마진↑, 회원 트래픽 필요 |
| 오픈마켓(쿠팡/11번가/스마트스토어) | 회전↑, 수수료·가격경쟁↑ |
| 도매 특판 | 대량 처분↑, 단가↓ |
| 오프라인 매장(공간 파트너) | 신뢰·체험·조합원 교육 결합 |
| 공동구매 | 단기 폭발 수요·유통기한 임박품 |

**배분 예시**
```
LG그램:  자체몰 30% / 도매특판 50% / 오픈마켓 20%
마스크팩: 자체몰 50% / 공동구매 30% / 오프라인 20%
식품류:  유통기한 짧음 → 오픈마켓/공동구매 우선(빠른 회전)
```

## 6) AI 역할 ④ — 가격 최적화 ("가격은 고정이 아니라 매일 바뀐다")

### 6-1. 추천 판매가 공식
```
추천 판매가 = 매입원가 + 물류비 + 플랫폼 수수료 + 목표 마진 + 위험보정값
```
가격 결정 입력: 현재 최저가 · 평균 판매가 · 경쟁 판매자 수 · 최근 7일 판매속도 ·
재고 잔량 · 체류일수 · 목표 마진율 · 수수료 · 배송비 · 반품률

### 6-2. 체류일 기반 마진 감쇠 (재고가 오래 머물면 마진을 낮춰 회전)
```
입고 1~7일차:   목표 마진 35%
입고 8~20일차:  목표 마진 25%
입고 21~35일차: 목표 마진 15%
입고 36일 이후: 특판/공동구매/묶음판매 전환
```
> 단, 긴급 처분에서도 `추천가 ≥ 원가 × (1+MIN_MARGIN)` 보장 — **손절이지 헐값 투매 금지.**
> 이것이 AI가 "돈 버는 마법"이 아니라 "손실을 줄이는 응급실 분류 간호사"로 작동하는 지점.

## 7) AI 역할 ⑤ — 재고 회전 위험 감지 (색상 트리아지)

재고는 시간이 지날수록 위험이 커지므로, SKU마다 체류 기간·판매율을 상시 감시합니다.

| 등급 | 체류일 | 상태 | AI 권고 |
|---|---|---|---|
| 🟢 Green | 0~14일 | 정상 회전 | 가격 유지 |
| 🟡 Yellow | 15~30일 | 판매 둔화 | 가격 조정 |
| 🟠 Orange | 31~45일 | 경쟁가 하락 | 특판 전환 |
| 🔴 Red | 45일+ | 긴급 처분 | 도매매각·묶음·프로모션 |

**전산 화면 예시**
```
상품명        체류일수  판매율  위험등급  AI 권고
LG그램        12일     68%    Green    자체몰 유지
청소기        28일     35%    Yellow   가격 7% 인하
마스크팩      39일     22%    Orange   공동구매 전환
식품세트      44일     18%    Red      특판 즉시 진행
```

**보조 위급도 점수(TUS)** — 체류일 외에 다차원 위급도를 더 정밀히 산출하려면
6신호(수요하락·경쟁가침식·체류일·재고깊이·시즌이탈·마진소멸)를 0~1 정규화해 가중합한
`TUS(0~100)`를 함께 계산하고, 원가 이하 경쟁 등장 시 강제 Red 승격시킵니다.
(가중치·임계는 `triage_config.json`으로 분리 — 색상 등급의 정밀 백업 지표로 사용.)

## 8) AI 역할 ⑥ — 정산 검증 (룰 엔진 우선 + AI 이상탐지)

정산은 **AI가 계산·결정하지 않습니다.** 반드시 **룰 엔진**이 계산하고, AI는 **오류를 감지**만 합니다.

### 8-1. 정산 룰 (결정론)
```
상품별 순마진 = 매출 − 상품원가 − 플랫폼수수료 − 배송비 − 물류비 − 반품/CS비용
정산 예정액   = 상품별 순마진 × 계약별 배분율
```

### 8-2. AI 이상탐지 (패턴 감지)
- 판매됐는데 출고 기록 없음
- 출고됐는데 매출 기록 없음
- 같은 SKU가 중복 배정됨
- 특정 채널 수수료가 평균보다 과도
- 반품률 급상승
- 정산액이 과거 패턴과 크게 상이

→ 이상 발견 시 정산을 자동 확정하지 않고 **담당자 검토 큐**로 보냅니다.

## 9) AI 판단 엔진 구조 — 단일 모델이 아니라 엔진 파이프라인

```
AI 상품 평가 엔진 (수요/마진/위험 점수)
        ↓
AI 매입 수량 엔진 (적정 수량 / 과잉재고 방지)
        ↓
AI 채널 배분 엔진 (자체몰 / 오픈마켓 / 특판)
        ↓
AI 가격 최적화 엔진 (인상 / 인하 / 유지)
        ↓
AI 정산 검증 엔진 (이상거래 / 누락 / 오류 감지)
```

### 사람 개입(Human-in-the-loop) 성숙 단계
초기에는 "AI 자동 실행"보다 "AI 추천 + MD 승인"이 안전합니다.

| 단계 | 방식 |
|---|---|
| 초기 | AI 추천 → MD 검토 → 승인 → 실행 |
| 고도화 | AI 추천 → 예외만 MD 검토 → 자동 실행 |
| 성숙 | 저위험 상품 자동 실행 / 고위험 상품 수동 승인 |

## 10) 단일 원장(Single Ledger) — "보이는 상품 · 보이는 창고 · 보이는 정산"

AI보다 더 중요한 것이 **원장**입니다. AI가 좋아도 정산 데이터가 분리돼 있으면 신뢰가 무너집니다.
모든 거래는 **하나의 append-only 원장** 구조로 저장합니다.

```
[계약 원장]   누가, 얼마를, 어떤 조건으로 참여했는가
[상품 원장]   어떤 SKU가, 누구의 운용금액과 매칭되었는가
[입고 원장]   언제, 어느 창고, 어느 랙에 입고되었는가
[판매 원장]   어느 채널에서, 얼마에, 몇 개 팔렸는가
[출고 원장]   언제, 누구에게, 어떤 송장으로 출고되었는가
[정산 원장]   매출·원가·수수료·물류비·마진·정산액은 얼마인가
[재매입 원장] 정산 후 어떤 상품으로 다시 운용되는가
```

- 각 이벤트에 `input_hash`(그날 입력 해시)를 남겨 **동일 입력 = 동일 판단** 재현·감사 가능.
- 조합원 앱은 "파주 창고 렉3-2열 립스틱이 어제 5만원 결제 → 오늘 출고" 로그를 실시간 동기화.

## 11) 데이터베이스 스키마 (개발 관점 최소 테이블)

핵심 키는 **`owner_contract_id`** — 조합원/도매파트너의 참여금이 어떤 실물 SKU·LOT와
연결되는지 추적하는 연결 고리입니다.

```
1. members        member_id, name, type, referrer_id, status
2. contracts      contract_id, member_id, package_amount, start_date, end_date,
                  auto_renewal, termination_condition
3. sku_master     sku_id, brand, product_name, category, barcode, model_no,
                  expiry_date, risk_type
4. inventory_lot  lot_id, sku_id, warehouse_id, rack_location, quantity_in,
                  quantity_available, quantity_sold, owner_contract_id   ★
5. market_price   sku_id, channel, collected_date, min_price, avg_price,
                  max_price, seller_count
6. ai_score       sku_id, demand_score, margin_score, risk_score,
                  rotation_score, recommended_action
7. sales_order    order_id, sku_id, channel, sale_price, quantity, fee,
                  shipping_cost, order_date
8. settlement     settlement_id, contract_id, revenue, cost, fee,
                  logistics_cost, gross_margin, settlement_amount, status
9. audit_log      event_id, actor, action, before_value, after_value, timestamp
```

- `market_price`는 2-2의 네이버/오픈마켓 수집값을 매일 적재 → 경쟁가 시계열의 원천.
- `ai_score`는 3~7절 엔진의 출력 스냅샷.
- `audit_log` + 정산/판매 원장이 10절 단일 원장을 구성.

## 17) 미결 결정 사항 (확정 필요)

1. **오픈마켓 커넥터**: 쿠팡/11번가는 공식 API·파트너 계약 전까지 크롤러 vs 수동 CSV 중 무엇으로?

5. **UI 형태**: 기존 `app.py` Streamlit 탭 통합 vs 독립 운영 콘솔(신규).
6. **자동화 진입점**: P5에서 "AI 추천 + MD 승인"만 구현하고 자동 실행은 후순위로 둘지.
